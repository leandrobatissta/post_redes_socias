Resumo

Este paper descreve a arquitetura, implementação e validação de um sistema completo de Kiosk/Loja virtual desenvolvido para operação autônoma ininterrupta (24/7) sem necessidade de intervenção manual. O sistema integra cinco componentes de software — quatro em produção e um em desenvolvimento — comunicando-se via protocolo serial proprietário. São implementadas quatro camadas de proteção: física, de software, de sistema e operacional. A fase v1.0 foi entregue em 7 dias por um único desenvolvedor, com aceleração significativa proporcionada pelo uso de ferramentas de Inteligência Artificial — uma tarefa que, no modelo tradicional com equipe, demandaria entre 30 e 60 dias.

Palavras-chave: Kiosk, Protocolo Serial Proprietário, Bloqueio Físico, Automação, Linux, Python, Inteligência Artificial, Desenvolvimento Acelerado.

1. Introdução

Sistemas de Kiosk são amplamente utilizados em aplicações comerciais que exigem operação contínua e segura sem supervisão humana direta. O desafio fundamental nesse domínio é garantir que o sistema opere de forma autônoma, resista a tentativas de manipulação física e digital, e se recupere automaticamente de falhas — tudo isso com latência mínima e sem interrupção do serviço.

O sistema foi concebido com uma base técnica sólida pré-existente — arquitetura, protocolos e criptografia desenvolvidos por Cleyton Pedroza — e implementado em sua totalidade funcional por Leandro Batista em 7 dias, com auxílio de ferramentas de IA. Este resultado representa uma redução de 75–90% no tempo de desenvolvimento estimado pelo modelo tradicional (30–60 dias com equipe).

Nota: Detalhes de configuração, caminhos de instalação, mensagens do protocolo e identificadores internos foram omitidos intencionalmente. O sistema encontra-se em fase de stage/teste em campo.

3. Protocolo de Comunicação Proprietário

Protocolo serial proprietário desenvolvido por Cleyton Pedroza para padronizar a troca de mensagens entre os componentes e os periféricos de hardware. Opera sobre interface serial física e define comandos para controle de sensores físicos, periféricos de entrada de valores e iluminação.

Nota de segurança: A estrutura de mensagens, identificadores de comandos, endereçamento e parâmetros do protocolo são informações proprietárias e não são divulgados neste documento.

Garantias implementadas: integridade por checksum por mensagem, timeout com prevenção de deadlocks, reconexão automática serial, monitoramento contínuo de componentes e isolamento total do sistema de bloqueio físico.

5. Sistema de Bloqueio Físico (System Lock)

5.1 Motivação

Em sistemas Kiosk de uso público, um vetor de ataque comum é a inserção de dispositivos de entrada externos para obter acesso ao sistema operacional subjacente. O System Lock mitiga esse vetor com seis camadas independentes de bloqueio no kernel Linux — garantindo que, mesmo que uma camada seja contornada, as demais continuem ativas.

5.2 Arquitetura em 6 Camadas Independentes

Camada 1 — Bloqueio Direto via Sistema de Arquivos (Prioridade Máxima)

Remoção completa de permissões de acesso ao dispositivo no sistema de arquivos Linux, combinada com desabilitamento via interface de controle do kernel. Maior prioridade e menor latência.

Camada 2 — Bloqueio via Camada Gráfica

Desabilitamento do dispositivo na camada de servidor gráfico do sistema. Garante que dispositivos com permissões de sistema não possam enviar eventos para aplicações gráficas.

Camada 3 — Remoção de Módulos do Kernel

Descarregamento seletivo dos módulos do kernel responsáveis pelo suporte a dispositivos de interface humana. Impede que o kernel processe eventos do dispositivo na raiz do suporte ao hardware.

Camada 4 — Bloqueio de Interface de Dispositivo (Wireless / Bluetooth)

Bloqueio específico para dispositivos que comunicam via interface de entrada de baixo nível, cobrindo wireless RF, dongles 2.4GHz e Bluetooth. Remoção de permissões nos nós de dispositivo correspondentes.

Camada 5 — Captura Exclusiva via Chamada de Sistema

Chamada de sistema para captura exclusiva do dispositivo. Apenas o processo detentor recebe os eventos, impedindo que qualquer outra aplicação processe entradas do dispositivo bloqueado.

Camada 6 — Regras Persistentes de Gerenciamento de Dispositivos

Regras no sistema de gerenciamento de dispositivos que garantem bloqueio persistente após reinicializações e aplicação automática a qualquer novo dispositivo inserido.

7. Sistema de Atualização (Updater)

  • Verificação de versão — Comparação entre versão local e versão disponível no servidor remoto.
  • Download seguro — Transferência via canal criptografado com verificação de autenticidade.
  • Backup da versão atual — Preservação completa antes de qualquer modificação.
  • Validação de integridade — Checksum criptográfico do artefato antes da instalação.
  • Instalação silenciosa — Sem interação do usuário e sem interrupção do serviço.
  • Teste pós-instalação — Verificação de funcionamento do componente atualizado.
  • Limpeza — Remoção de arquivos temporários.
  • Rollback automático — Em caso de falha em qualquer passo, restauração da versão anterior.

12. Conclusão

Este paper apresentou a arquitetura e implementação de um sistema Kiosk com operação autônoma 24/7, proteção multi-camada e automação completa de ciclo de vida. O sistema demonstra que é possível construir infraestrutura de software embarcado robusta e de produção combinando três elementos fundamentais:

  1. Arquitetura bem definida como fundação — concepção técnica sólida que define protocolos, fluxos e interfaces (contribuição de Cleyton Pedroza)
  2. Implementação focada com conhecimento de domínio — transformação da arquitetura em produto funcional, testado e em produção (contribuição de Leandro Batista)
  3. IA como acelerador de execução — compressão do ciclo de 30–60 dias para 7 dias, mantendo qualidade técnica
